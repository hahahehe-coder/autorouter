# AutoRouter

> **让合适的任务，自动交给合适的模型。**

## 别再为每个问题手动挑选模型

简单的问题，用不着最昂贵的模型；复杂的任务，也不该交给能力不足的模型。

但在实际使用中，我们常常只能凭经验反复切换：日常问答选便宜模型，遇到编程和复杂推理再换强模型，任务变了还要重新选择。模型越来越多，这件事也越来越麻烦。

AutoRouter 就是为了解决这个问题。

它会分析每次请求的内容和任务难度，自动选择更合适的模型：

- 日常问答交给快速、实惠的模型
- 编程任务自动切换到更擅长代码的模型
- 复杂推理调用能力更强的模型
- 图片和长上下文请求自动匹配具备相应能力的模型

你不再需要提前判断任务有多难，也不用在多个模型之间来回切换。

**你只管提问，模型选择交给 AutoRouter。**

## 少花不必要的钱，也不让复杂任务将就

如果所有请求都调用最强模型，当然省心，但大量简单任务也会产生不必要的费用。

如果始终使用便宜模型，成本降低了，遇到编程、推理和复杂任务时，回答效果又可能不够理想。

AutoRouter 会根据任务难度动态分配模型，让简单任务优先使用高性价比模型，让真正困难的任务使用更强的模型。在使用效果和 API 成本之间，取得更合理的平衡。

> **省掉不必要的模型成本，不让真正困难的任务降级。**

## 一个入口，连接多个模型

AutoRouter 位于你的客户端和上游 API 服务之间。客户端只需要连接 AutoRouter，后续请求由它自动完成分析、选模和转发。

模型可以自由配置，路由策略也可以根据实际需求调整。无论你使用的是 API 中转服务，还是拥有多个模型的兼容供应商，都可以把不同模型组织成一套自动路由方案。

一次配置，之后每次请求都能自动选择。

## API 站长：增加一个渠道，就能提供自动选模

将 AutoRouter 部署到服务器，再把它作为一个渠道接入现有 API 中转站，即可为用户增加一个能够自动选择模型的入口。

用户不需要了解背后配置了多少模型，也不需要自己判断应该调用哪个模型。只需选择 AutoRouter 对应的渠道或模型名称，系统就会根据任务内容完成后续路由。

这意味着你可以：

- 把多个不同价格、不同能力的模型组合成统一服务
- 减少简单请求对高价模型的浪费
- 为复杂任务保留更强的模型能力
- 在不改变用户使用习惯的情况下增加智能路由

部署一次，就能让现有模型池更灵活地工作。

## 普通用户：在本机启动，现有客户端照常使用

不经营 API 中转站，也可以使用 AutoRouter。

在电脑上启动服务，把聊天客户端或开发工具的 API 地址指向 AutoRouter，就可以继续使用原来的客户端。以后不论是日常问答、写代码还是复杂分析，都不必再手动更换模型。

你看到的是一个统一入口，背后的模型选择由 AutoRouter 自动完成。

## 不只是分类，还有完整的路由保护

AutoRouter 借鉴 OpenSquilla 的模型路由设计，并结合实际 API 使用场景加入多层路由策略。除了判断任务难度，它还会综合考虑：

- 模型路由结果及置信度
- 编程、推理等任务特征
- 当前请求是否包含图片
- 上下文长度是否超过模型能力
- 同一会话是否应该保持较高档位
- 低置信度情况下是否需要使用更可靠的模型

当 ML 路由不可用时，系统还可以自动回退到启发式规则，避免因为分类组件异常而影响正常请求。

## 按你的需求，自由配置

你可以在管理界面中配置：

- 可用模型及其上下文、视觉能力
- 不同难度对应的模型档位
- 最大输出长度和思考强度
- 大上下文与图片请求的升级策略
- 上游服务地址和连接信息
- ML 路由和置信度阈值

配置修改后可以立即重载，无需反复重启服务。

## 从此，不必再纠结该选哪个模型

模型的数量还会继续增加，但你不需要为每一次请求重新做选择。

让便宜模型处理它擅长的简单任务，让强模型专注于真正困难的问题。既减少不必要的开销，也保留复杂任务需要的能力。

**一个入口，多个模型，自动选择。**

**AutoRouter——你只管提问，剩下的交给路由。**
